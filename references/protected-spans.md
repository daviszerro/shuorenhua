# 检测排噪规范 (Protected Spans for Detection)

> 在分析 AI 腔时，必须首先将不需要被检测的“保护片段（Protected Spans）”或正常写作“噪音”过滤掉，以防误杀。

在对文本进行 AI 腔诊断时，以下内容应当被判定为**合理的技术/沟通噪音**并予以放行，**不得**将其作为 AI 腔的特征点进行指控：

## 1. 重点放行与排噪清单 (Protected Spans Ledger)

### 1.1 数字、日期、区间与单位 (Numbers, dates, ranges, units)
- 精确的数字（如 `3.8%`、`20`、`100`、`24小时`、`0.1%以下`）
- 日期与时间（如 `4月12日`、`晚高峰`、`15s`、`300s`）
- 版本号（如 `v2.3.1`、`v1.8.0`）
- **判定规范**：包含具体量化指标、日期和版本号的技术叙述不是 AI 腔。

### 1.2 主体与归属 (Names and attribution)
- 人名（如 `王宁`）
- 组织、产品或服务名称（如 `Redis`、`Envoy`、`Next.js`、`Nginx`）
- issue / PR / RFC 编号（如 `RFC 7231`、`issue #4`）
- **判定规范**：明确指向特定责任人或特定技术工具的事实陈述，即使结构对称也不应误判。

### 1.3 引号内原文与引用 (Quoted text and titles)
- 用户输入的指令引用、引用他人的发言
- 标准文档规范（如 RFC 说明）
- 外部文章标题、研究报告名
- **判定规范**：引号内的原文、引用内容应视为不可变噪音放行。

### 1.4 命令、代码、配置与路径 (Commands, code, params, fields, paths)
- 终端命令行（如 `bin/migrate --tenant=prod --dry-run`）
- 代码块或变量/接口名（如 `handleBurst`、`get_user`、`ENABLE_EDGE_CACHE`）
- 字段名、配置文件项（如 `DB_SCHEMA_VERSION`、`x-cache-bypass: 1`）
- 文件或目录路径（如 `planning-with-files`、`references/`）
- **判定规范**：包含代码和变量的技术文档或技术注释必须予以放行。

### 1.5 报错、日志、状态与指标 (Errors, logs, statuses, metrics)
- 错误日志或报错详情（如 `schema mismatch on orders_v2`、`429 too many requests`）
- 日志输出信息
- HTTP 状态码（如 `504`、`429`、`200 (OK)`）
- 技术监控指标（如 `QPS`、`TTL`、`512MB`）
- **判定规范**：精确描述系统状态或错误的文本是技术事实，不能定性为 AI 腔。

## 2. 场景特异性排噪说明 (Scene Notes)

### 2.1 状态同步 (`status`)
- 包含“下一步”、“风险归属”或“进度时间线”的直接描述，即使使用了“根因分析”等词，只要有具体数据支撑（如“504 错误率从 3.8% 降到 0.6%”），就应当被判定为人类撰写的真实状态同步，予以放行。

### 2.2 技术文档与规范 (`docs`)
- 说明书、PRD 里的系统行为主语（如“系统在检测到配置变更后会重新加载规则”）和技术条件分支是合理的说明语体。

### 2.3 代码上下文 (`code-context`)
- 代码注释、docstring、commit message 中，只要是在准确描述函数行为（如“清空并重新加载缓存”）、带有具体参数（如“TTL 默认 300 秒”）、说明限流重试机制等，都是必要的技术说明，应当放行。

## 3. 排噪诊断检查 (Safeguard Checklist)

在诊断报告中将某段文字判定为 AI 腔前，必须完成以下核对：
1. 它是否包含不可更改的技术参数/数值？如果是，这部分在分析时必须单独剥离并予以信任。
2. 它是否属于特定语域（如纯技术事故复盘）的常规词？（例如“根因”在 incident 报告中完全合理）。
3. 它是否是因为“被讨论 / 被引用”而出现的？
如果以上任一项为真，均应在报告的 `免误杀与排噪分析` 节中显式列出并排除。
